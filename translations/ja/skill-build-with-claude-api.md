# Claudeを使用したLLMパワードアプリケーションの構築

このスキルは、Claudeを使用したLLMパワードアプリケーションの構築を支援します。ニーズに基づいて適切なサーフェスを選択し、プロジェクト言語を検出してから、関連する言語固有のドキュメントを読んでください。

## デフォルト設定

ユーザーが別途要求しない限り：

Claudeモデルバージョンについては、{{OPUS_NAME}}を使用してください。正確なモデル文字列`{{OPUS_ID}}`でアクセスできます。少しでも複雑なものにはアダプティブシンキング（`thinking: {type: "adaptive"}`）をデフォルトで使用してください。最後に、長い入力、長い出力、または高い`max_tokens`を伴う可能性のあるリクエストにはストリーミングをデフォルトで使用してください — これによりリクエストタイムアウトを防ぎます。個々のストリームイベントを処理する必要がない場合は、SDKの`.get_final_message()` / `.finalMessage()`ヘルパーを使用して完全なレスポンスを取得してください。

## 言語検出

コード例を読む前に、ユーザーが使用している言語を判断します：

1. **プロジェクトファイルを確認**して言語を推測：

   - `*.py`、`requirements.txt`、`pyproject.toml`、`setup.py`、`Pipfile` → **Python** — `python/`から読む
   - `*.ts`、`*.tsx`、`package.json`、`tsconfig.json` → **TypeScript** — `typescript/`から読む
   - `*.js`、`*.jsx`（`.ts`ファイルなし） → **TypeScript** — JSは同じSDKを使用、`typescript/`から読む
   - `*.java`、`pom.xml`、`build.gradle` → **Java** — `java/`から読む
   - `*.kt`、`*.kts`、`build.gradle.kts` → **Java** — KotlinはJava SDKを使用、`java/`から読む
   - `*.scala`、`build.sbt` → **Java** — ScalaはJava SDKを使用、`java/`から読む
   - `*.go`、`go.mod` → **Go** — `go/`から読む
   - `*.rb`、`Gemfile` → **Ruby** — `ruby/`から読む
   - `*.cs`、`*.csproj` → **C#** — `csharp/`から読む
   - `*.php`、`composer.json` → **PHP** — `php/`から読む

2. **複数の言語が検出された場合**（例：PythonとTypeScriptの両方のファイル）：

   - ユーザーの現在のファイルまたは質問がどの言語に関連するか確認
   - それでも曖昧な場合は質問：「PythonとTypeScriptの両方のファイルを検出しました。Claude API統合にはどの言語を使用していますか？」

3. **言語が推測できない場合**（空のプロジェクト、ソースファイルなし、またはサポートされていない言語）：

   - AskUserQuestionでオプションを提示：Python、TypeScript、Java、Go、Ruby、cURL/生HTTP、C#、PHP
   - AskUserQuestionが利用できない場合は、Pythonの例をデフォルトとし、「Pythonの例を表示しています。別の言語が必要な場合はお知らせください。」と注記

4. **サポートされていない言語が検出された場合**（Rust、Swift、C++、Elixirなど）：

   - `curl/`からcURL/生HTTPの例を提案し、コミュニティSDKが存在する可能性があることを注記
   - 参考実装としてPythonまたはTypeScriptの例を提示

5. **ユーザーがcURL/生HTTPの例を必要とする場合**、`curl/`から読みます。

## どのサーフェスを使用すべきか？

> **シンプルに始めましょう。** ニーズを満たす最もシンプルな階層をデフォルトにしてください。単一APIコールとワークフローはほとんどのユースケースに対応します — タスクが本当にオープンエンドでモデル駆動の探索を必要とする場合にのみエージェントを使用してください。

| ユースケース                                    | 階層            | 推奨サーフェス            | 理由                                    |
| ----------------------------------------------- | --------------- | ------------------------- | --------------------------------------- |
| 分類、要約、抽出、Q&A                           | 単一LLMコール   | **Claude API**            | 1リクエスト、1レスポンス                |
| バッチ処理または埋め込み                        | 単一LLMコール   | **Claude API**            | 専用エンドポイント                      |
| コード制御ロジックによるマルチステップパイプライン | ワークフロー    | **Claude API + ツール使用** | ループはユーザーが制御                  |
| 独自ツールを持つカスタムエージェント            | エージェント    | **Claude API + ツール使用** | 最大の柔軟性                            |
| ファイル/ウェブ/ターミナルアクセスを持つAIエージェント | エージェント | **エージェントSDK**       | 組み込みツール、安全性、MCPサポート     |
| エージェント型コーディングアシスタント          | エージェント    | **エージェントSDK**       | このユースケース向けに設計              |
| 組み込みの権限とガードレールが必要              | エージェント    | **エージェントSDK**       | 安全機能が含まれる                      |

> **注意：** エージェントSDKは、組み込みのファイル/ウェブ/ターミナルツール、権限、MCPをすぐに使いたい場合に適しています。独自のツールでエージェントを構築したい場合は、Claude APIが正しい選択です — 自動ループ処理にはツールランナーを使用し、細かい制御（承認ゲート、カスタムロギング、条件付き実行）には手動ループを使用してください。

### 決定ツリー

```
アプリケーションに何が必要ですか？

1. 単一LLMコール（分類、要約、抽出、Q&A）
   └── Claude API — 1リクエスト、1レスポンス

2. Claudeはその作業の一部としてファイルの読み書き、ウェブ閲覧、または
   シェルコマンドの実行が必要ですか？（あなたのアプリがファイルを読んで
   Claudeに渡すのではなく — Claude自身がファイル/ウェブ/シェルを
   発見してアクセスする必要がありますか？）
   └── はい → エージェントSDK — 組み込みツール、再実装不要
       例：「コードベースをスキャンしてバグを見つける」、「ディレクトリ内のすべてのファイルを要約する」、
            「サブエージェントを使用してバグを見つける」、「ウェブ検索でトピックを調査する」

3. ワークフロー（マルチステップ、コードオーケストレーション、独自ツール）
   └── ツール使用を伴うClaude API — ループはユーザーが制御

4. オープンエンドエージェント（モデルが独自の軌道を決定、独自ツール）
   └── Claude APIエージェントループ（最大の柔軟性）
```

### エージェントを構築すべきか？

エージェント階層を選択する前に、4つの基準すべてを確認してください：

- **複雑さ** — タスクはマルチステップで、事前に完全に指定するのが難しいですか？（例：「この設計ドキュメントをPRにする」vs「このPDFからタイトルを抽出する」）
- **価値** — 結果は高いコストとレイテンシを正当化しますか？
- **実現可能性** — Claudeはこのタスクタイプが得意ですか？
- **エラーのコスト** — エラーを検出して回復できますか？（テスト、レビュー、ロールバック）

これらのいずれかに「いいえ」の場合は、よりシンプルな階層（単一コールまたはワークフロー）にとどまってください。

## アーキテクチャ

すべてが`POST /v1/messages`を通過します。ツールと出力制約はこの単一エンドポイントの機能であり、別個のAPIではありません。

**ユーザー定義ツール** — ツールを定義し（デコレータ、Zodスキーマ、または生JSONを介して）、SDKのツールランナーがAPIの呼び出し、関数の実行、Claudeが完了するまでのループを処理します。完全な制御が必要な場合は、ループを手動で記述できます。

**サーバーサイドツール** — Anthropicのインフラストラクチャで実行されるAnthropicホスト型ツール。コード実行は完全にサーバーサイドです（`tools`で宣言すると、Claudeが自動的にコードを実行）。コンピュータ使用はサーバーホスト型またはセルフホスト型が可能です。

**構造化出力** — Messages APIのレスポンス形式（`output_config.format`）やツールパラメータ検証（`strict: true`）を制約します。推奨アプローチは`client.messages.parse()`で、スキーマに対するレスポンスを自動的に検証します。注意：古い`output_format`パラメータは非推奨です。`messages.create()`では`output_config: {format: {...}}`を使用してください。

**サポートエンドポイント** — バッチ（`POST /v1/messages/batches`）、ファイル（`POST /v1/files`）、トークンカウントはMessages APIリクエストにフィードまたはサポートします。

| ----------------- | ------------------- | -------------- | --------- | --------- |
| Claude Opus 4.6   | `claude-opus-4-6`   | 200K（1Mベータ）| $5.00     | $25.00    |
| Claude Sonnet 4.6 | `claude-sonnet-4-6` | 200K（1Mベータ）| $3.00     | $15.00    |

**ユーザーが明示的に別のモデルを指定しない限り、常に`{{OPUS_ID}}`を使用してください。** これは交渉の余地がありません。ユーザーが文字通り「sonnetを使って」または「haikuを使って」と言わない限り、`{{SONNET_ID}}`、`{{PREV_SONNET_ID}}`、またはその他のモデルを使用しないでください。コストのためにダウングレードしないでください — それはユーザーの判断であり、あなたの判断ではありません。

**重要：上の表のモデルID文字列のみを正確に使用してください — それらはそのまま完全です。日付サフィックスを追加しないでください。** 例えば、`claude-sonnet-4-5`を使用し、`claude-sonnet-4-5-20250514`やトレーニングデータから記憶している他の日付サフィックス付きバリアントは決して使用しないでください。ユーザーが表にない古いモデルを要求した場合（例：「opus 4.5」、「sonnet 3.7」）、正確なIDについては`shared/models.md`を読んでください — 自分で構築しないでください。

注意：上記のモデル文字列のいずれかが見慣れない場合、それは予想通りです — それはトレーニングデータのカットオフ後にリリースされたことを意味します。それらは実際のモデルです。私たちはそのようなことであなたをだましたりしません。

## シンキングとエフォート（クイックリファレンス）

**Opus 4.6 — アダプティブシンキング（推奨）：** `thinking: {type: "adaptive"}`を使用します。Claudeはいつ、どのくらい考えるかを動的に決定します。`budget_tokens`は不要です — `budget_tokens`はOpus 4.6とSonnet 4.6で非推奨であり、使用してはいけません。アダプティブシンキングは自動的にインターリーブシンキングも有効にします（ベータヘッダー不要）。**ユーザーが「拡張シンキング」、「シンキングバジェット」、または`budget_tokens`を求めた場合：常にOpus 4.6で`thinking: {type: "adaptive"}`を使用してください。シンキング用の固定トークンバジェットの概念は非推奨です — アダプティブシンキングがそれに置き換わります。`budget_tokens`を使用しないでください。また、古いモデルに切り替えないでください。**

**エフォートパラメータ（GA、ベータヘッダー不要）：** `output_config: {effort: "low"|"medium"|"high"|"max"}`（トップレベルではなく`output_config`内）を介してシンキングの深さと全体的なトークン消費を制御します。デフォルトは`high`（省略した場合と同等）。`max`はOpus 4.6のみ。Opus 4.5、Opus 4.6、Sonnet 4.6で動作します。Sonnet 4.5 / Haiku 4.5ではエラーになります。最良のコスト品質トレードオフのためにアダプティブシンキングと組み合わせてください。サブエージェントや単純なタスクには`low`を、最も深い推論には`max`を使用してください。

**Sonnet 4.6：** アダプティブシンキング（`thinking: {type: "adaptive"}`）をサポートします。`budget_tokens`はSonnet 4.6で非推奨です — 代わりにアダプティブシンキングを使用してください。

**古いモデル（明示的に要求された場合のみ）：** ユーザーが特にSonnet 4.5または別の古いモデルを要求した場合は、`thinking: {type: "enabled", budget_tokens: N}`を使用してください。`budget_tokens`は`max_tokens`より小さくなければなりません（最小1024）。ユーザーが`budget_tokens`に言及したからといって古いモデルを選択しないでください — 代わりにOpus 4.6でアダプティブシンキングを使用してください。

## コンパクション（クイックリファレンス）

**ベータ、Opus 4.6のみ。** 200Kコンテキストウィンドウを超える可能性のある長時間の会話の場合、サーバーサイドコンパクションを有効にします。APIはトリガー閾値（デフォルト：150Kトークン）に近づくと、以前のコンテキストを自動的に要約します。ベータヘッダー`compact-2026-01-12`が必要です。

**重要：** 各ターンで`response.content`（テキストだけでなく）をメッセージに追加してください。レスポンス内のコンパクションブロックを保持する必要があります — APIは次のリクエストでコンパクト化された履歴を置き換えるためにそれらを使用します。テキスト文字列のみを抽出して追加すると、コンパクション状態が静かに失われます。

コード例については`{lang}/claude-api/README.md`（コンパクションセクション）を参照してください。完全なドキュメントは`shared/live-sources.md`のWebFetch経由で取得できます。

## 一般的な落とし穴

- APIにファイルやコンテンツを渡す際に入力を切り詰めないでください。コンテンツがコンテキストウィンドウに収まらないほど長い場合は、ユーザーに通知し、静かに切り詰めるのではなく、オプション（チャンク化、要約など）を話し合ってください。
- **Opus 4.6 / Sonnet 4.6のシンキング：** `thinking: {type: "adaptive"}`を使用してください — `budget_tokens`を使用しないでください（Opus 4.6とSonnet 4.6の両方で非推奨）。古いモデルでは、`budget_tokens`は`max_tokens`より小さくなければなりません（最小1024）。これを間違えるとエラーがスローされます。
- **Opus 4.6のプリフィル削除：** アシスタントメッセージのプリフィル（最後のアシスタントターンのプリフィル）はOpus 4.6で400エラーを返します。レスポンス形式を制御するには、代わりに構造化出力（`output_config.format`）またはシステムプロンプトの指示を使用してください。
- **128K出力トークン：** Opus 4.6は最大128Kの`max_tokens`をサポートしますが、SDKはHTTPタイムアウトを避けるために大きな`max_tokens`にはストリーミングが必要です。`.stream()`と`.get_final_message()` / `.finalMessage()`を使用してください。
- **ツールコールのJSON解析（Opus 4.6）：** Opus 4.6はツールコールの`input`フィールドで異なるJSONの文字列エスケープ（例：Unicodeまたはフォワードスラッシュエスケープ）を生成する場合があります。常にツール入力を`json.loads()` / `JSON.parse()`で解析してください — シリアライズされた入力で生の文字列マッチングを行わないでください。
- **構造化出力（すべてのモデル）：** `messages.create()`では非推奨の`output_format`パラメータの代わりに`output_config: {format: {...}}`を使用してください。これは4.6固有ではなく、一般的なAPI変更です。
- **SDK機能を再実装しないでください：** SDKは高レベルヘルパーを提供しています — ゼロから構築するのではなく、それらを使用してください。具体的には：`.on()`イベントを`new Promise()`でラップする代わりに`stream.finalMessage()`を使用し、エラーメッセージの文字列マッチングの代わりに型付き例外クラス（`Anthropic.RateLimitError`など）を使用し、同等のインターフェースを再定義する代わりにSDK型（`Anthropic.MessageParam`、`Anthropic.Tool`、`Anthropic.Message`など）を使用してください。
- **SDKデータ構造用のカスタム型を定義しないでください：** SDKはすべてのAPIオブジェクトの型をエクスポートします。メッセージには`Anthropic.MessageParam`、ツール定義には`Anthropic.Tool`、ツール結果には`Anthropic.ToolUseBlock` / `Anthropic.ToolResultBlockParam`、レスポンスには`Anthropic.Message`を使用してください。独自の`interface ChatMessage { role: string; content: unknown }`を定義すると、SDKがすでに提供しているものを複製し、型安全性が失われます。
- **レポートとドキュメント出力：** レポート、ドキュメント、または可視化を生成するタスクでは、コード実行サンドボックスに`python-docx`、`python-pptx`、`matplotlib`、`pillow`、`pypdf`がプリインストールされています。Claudeはフォーマットされたファイル（DOCX、PDF、チャート）を生成し、Files API経由で返すことができます — プレーンなstdoutテキストの代わりに、「レポート」または「ドキュメント」タイプのリクエストにはこれを検討してください。
