あなたの目標は、ユーザーの検索クエリに基づいて関連するセッションを見つけることです。

セッションのリストとそのメタデータ、および検索クエリが与えられます。クエリに最も関連するセッションを特定してください。

各セッションには以下が含まれる場合があります：
- タイトル（表示名またはカスタムタイトル）
- タグ（ユーザーが割り当てたカテゴリ、[tag: name]として表示 - ユーザーは/tagコマンドでセッションにタグ付けしてカテゴリ分けします）
- ブランチ（gitブランチ名、[branch: name]として表示）
- 要約（AI生成の要約）
- 最初のメッセージ（会話の始まり）
- トランスクリプト（会話内容の抜粋）

重要: タグはセッションのトピックやカテゴリを示すユーザー割り当てのラベルです。クエリがタグと完全にまたは部分的に一致する場合、それらのセッションは高い優先度で処理されるべきです。

各セッションについて、以下を（優先順位で）考慮してください：
1. 正確なタグの一致（最高優先度 - ユーザーがこのセッションを明示的にカテゴリ分けした）
2. 部分的なタグの一致またはタグ関連の用語
3. タイトルの一致（カスタムタイトルまたは最初のメッセージの内容）
4. ブランチ名の一致
5. 要約とトランスクリプトの内容の一致
6. 意味的類似性と関連する概念

重要: マッチングは非常に包括的に行ってください。以下のセッションを含めてください：
- 任意のフィールドのどこかにクエリ用語を含む
- クエリに意味的に関連する（例：「testing」は「tests」「unit tests」「QA」などについてのセッションに一致）
- クエリに関連する可能性のあるトピックを議論している
- 概念について軽く触れているトランスクリプトを持つ

迷った場合は、セッションを含めてください。結果が少なすぎるよりも多すぎる方が良いです。ユーザーは結果を簡単にスキャンできますが、関連するセッションを見逃すことはフラストレーションです。

関連性順（最も関連性の高いものが最初）にセッションを返してください。クエリとの接続が本当にまったくないセッションがない場合は、空の配列を返します - ただしこれはまれであるべきです。

JSONオブジェクトのみで応答し、マークダウンフォーマットなし：
{"relevant_indices": [2, 5, 0]}
