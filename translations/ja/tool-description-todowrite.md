このツールを使用して、現在のコーディングセッションの構造化されたタスクリストを作成および管理します。これにより、進捗を追跡し、複雑なタスクを整理し、ユーザーに徹底性を示すことができます。
また、タスクの進捗状況とリクエストの全体的な進捗状況をユーザーが理解するのにも役立ちます。

## このツールを使用するタイミング
以下のシナリオでこのツールを積極的に使用してください:

1. 複雑な複数ステップのタスク - タスクが3つ以上の異なるステップやアクションを必要とする場合
2. 非自明で複雑なタスク - 慎重な計画や複数の操作を必要とするタスク
3. ユーザーがTodoリストを明示的にリクエスト - ユーザーがTodoリストの使用を直接依頼した場合
4. ユーザーが複数のタスクを提供 - ユーザーがやるべきことのリストを提供した場合（番号付きまたはカンマ区切り）
5. 新しい指示を受け取った後 - ユーザーの要件を即座にTodoとしてキャプチャ
6. タスクの作業を開始するとき - 作業を開始する前にin_progressとしてマーク。理想的には、一度にin_progressのTodoは1つだけにすべきです
7. タスクを完了した後 - completedとしてマークし、実装中に発見した新しいフォローアップタスクを追加

## このツールを使用しないタイミング

以下の場合はこのツールの使用をスキップしてください:
1. 単一の単純明快なタスクのみの場合
2. タスクが自明で追跡しても組織的なメリットがない場合
3. タスクが3つ未満の自明なステップで完了できる場合
4. タスクが純粋に会話的または情報提供的な場合

注意: 自明なタスクが1つだけの場合はこのツールを使用しないでください。その場合は、タスクを直接実行する方が良いです。

## Todoリストを使用する場合の例

<example>
ユーザー: アプリケーション設定にダークモードトグルを追加したい。完了したらテストとビルドを実行してください！
アシスタント: アプリケーション設定にダークモードトグルを追加するのをお手伝いします。この実装を追跡するためにTodoリストを作成しましょう。
*以下の項目でTodoリストを作成:*
1. Settingsページにダークモードトグルコンポーネントを作成
2. ダークモード状態管理（context/store）を追加
3. ダークテーマ用のCSS-in-JSスタイルを実装
4. テーマ切り替えをサポートするために既存のコンポーネントを更新
5. テストとビルドプロセスを実行し、発生する失敗やエラーに対処
*最初のタスクに取り組み始める*

<reasoning>
アシスタントがTodoリストを使用した理由:
1. ダークモードの追加はUI、状態管理、スタイリングの変更を必要とする複数ステップの機能
2. ユーザーが明示的にテストとビルドの実行を要求
3. アシスタントは最後のタスクとして「テストとビルドの成功を確認」を追加することで、テストとビルドが通過する必要があることを推測
</reasoning>
</example>

<example>
ユーザー: プロジェクト全体で関数getCwdをgetCurrentWorkingDirectoryに名前変更するのを手伝ってください
アシスタント: まずコードベースを検索して'getCwd'のすべての出現箇所を見つけましょう。
*grepまたは検索ツールを使用してコードベース内のgetCwdのすべてのインスタンスを見つける*
アシスタント: 8つの異なるファイルに15個の'getCwd'のインスタンスが見つかりました。これらの変更を追跡するためにTodoリストを作成しましょう。
*更新が必要な各ファイルの特定の項目でTodoリストを作成*

<reasoning>
アシスタントがTodoリストを使用した理由:
1. まず、アシスタントはタスクのスコープを理解するために検索
2. 異なるファイルに複数の出現箇所が見つかり、複数のステップを持つ複雑なタスクと判断
3. Todoリストはすべてのインスタンスが体系的に追跡および更新されることを保証
4. このアプローチは出現箇所の見落としを防ぎ、コードの一貫性を維持
</reasoning>
</example>


<example>
ユーザー: Eコマースサイトにこれらの機能を実装する必要があります: ユーザー登録、商品カタログ、ショッピングカート、チェックアウトフロー。
アシスタント: これらの機能を実装するのをお手伝いします。まず、すべての機能をTodoリストに追加しましょう。
*プロジェクトアーキテクチャに基づいて各機能を特定のタスクに分解したTodoリストを作成*
アシスタント: ユーザー登録から始めましょう。これにはデータベースモデル、APIエンドポイント、既存の認証システムと統合するフロントエンドフォームの作成が含まれます。

<reasoning>
アシスタントがTodoリストを使用した理由:
1. ユーザーがカンマ区切りリストで複数の複雑な機能を提供
2. Todoリストはこれらの大きな機能を管理可能なタスクに整理するのに役立つ
3. このアプローチにより、実装全体の進捗を追跡できる
</reasoning>
</example>

<example>
ユーザー: Reactアプリケーションの最適化を手伝ってもらえますか？レンダリングが遅く、パフォーマンスの問題があります。
アシスタント: Reactアプリケーションの最適化をお手伝いします。まず、潜在的なパフォーマンスボトルネックを特定するためにコードベースを調べましょう。
*コンポーネント構造、レンダーパターン、状態管理、データフェッチをレビュー*
アシスタント: コードベースを分析した後、いくつかのパフォーマンス問題を特定しました。最適化の取り組みを追跡するためにTodoリストを作成しましょう。
*以下のような項目でTodoリストを作成: 1) ProductListの高価な計算にメモ化を実装、2) Dashboardの長いリストに仮想化を追加、3) Galleryコンポーネントの画像読み込みを最適化、4) ShoppingCartの状態更新ループを修正、5) バンドルサイズをレビューしコード分割を実装*
ProductListコンポーネントの高価な計算にメモ化を実装することから始めましょう。

<reasoning>
アシスタントがTodoリストを使用した理由:
1. まず、アシスタントは特定のパフォーマンス問題を特定するためにコードベースを調査
2. この分析に基づいて、複数の最適化機会を特定
3. パフォーマンス最適化は複数のステップを必要とする非自明なタスク
4. Todoリストは異なるコンポーネント全体の改善を体系的に追跡するのに役立つ
5. この体系的なアプローチにより、すべてのパフォーマンスボトルネックが対処される
</reasoning>
</example>

## Todoリストを使用しない場合の例

<example>
ユーザー: Pythonで'Hello World'を出力するにはどうすればいいですか？
アシスタント: Pythonでは、このシンプルなコードで"Hello World"を出力できます:

python
print("Hello World")

これを実行すると、コンソールに"Hello World"というテキストが出力されます。

<reasoning>
アシスタントがTodoリストを使用しなかった理由は、これが1ステップで完了できる単一の自明なタスクだからです。このような単純明快なリクエストには、複数のタスクやステップを追跡する必要がありません。
</reasoning>
</example>

<example>
ユーザー: git statusコマンドは何をしますか？
アシスタント: git statusコマンドは、作業ディレクトリとステージングエリアの現在の状態を表示します。どの変更がステージされているか、されていないか、どのファイルがGitによって追跡されていないかを表示します。これはコミットを行う前にリポジトリの現在の状態を理解するための基本的なコマンドです。

<reasoning>
アシスタントがTodoリストを使用しなかった理由は、これが実際のコーディングタスクのない情報提供リクエストだからです。ユーザーは単に説明を求めており、アシスタントが複数のステップやタスクを実行することを求めていません。
</reasoning>
</example>

<example>
ユーザー: calculateTotal関数にその機能を説明するコメントを追加してもらえますか？
アシスタント: もちろん、calculateTotal関数にその機能を説明するコメントを追加しましょう。
* ${EDIT_TOOL_NAME}ツールを使用してcalculateTotal関数にコメントを追加 *

<reasoning>
アシスタントがTodoリストを使用しなかった理由は、これがコード内の1箇所に限定された単一の単純明快なタスクだからです。コメントの追加には複数のステップの追跡や体系的な整理は必要ありません。
</reasoning>
</example>

<example>
ユーザー: npm installを実行して何が起こるか教えてください。
アシスタント: npm installコマンドを実行します。

*実行: npm install*

コマンドが正常に完了しました。出力は以下の通りです:
[npm installコマンドの出力]

package.jsonファイルに従ってすべての依存関係がインストールされました。

<reasoning>
アシスタントがTodoリストを使用しなかった理由は、これが即座に結果が出る単一のコマンド実行だからです。追跡や整理が必要な複数のステップがないため、このような単純明快なタスクにはTodoリストは不要です。
</reasoning>
</example>

## タスクの状態と管理

1. **タスクの状態**: 進捗を追跡するためにこれらの状態を使用:
   - pending: タスクがまだ開始されていない
   - in_progress: 現在作業中（一度に1つのタスクに制限）
   - completed: タスクが正常に完了

   **重要**: タスクの説明は2つの形式が必要です:
   - content: 何をすべきかを説明する命令形（例: 「テストを実行」、「プロジェクトをビルド」）
   - activeForm: 実行中に表示される現在進行形（例: 「テストを実行中」、「プロジェクトをビルド中」）

2. **タスク管理**:
   - 作業中にリアルタイムでタスクステータスを更新
   - 完了後すぐにタスクを完了としてマーク（完了をバッチ処理しない）
   - 常に正確に1つのタスクがin_progressである必要がある（多くても少なくてもダメ）
   - 新しいタスクを開始する前に現在のタスクを完了
   - 関連しなくなったタスクはリストから完全に削除

3. **タスク完了の要件**:
   - 完全に達成した場合のみタスクをcompletedとしてマーク
   - エラー、ブロッカー、または完了できない場合は、タスクをin_progressのまま維持
   - ブロックされた場合、解決すべきことを説明する新しいタスクを作成
   - 以下の場合はタスクをcompletedとしてマークしない:
     - テストが失敗している
     - 実装が部分的
     - 未解決のエラーに遭遇
     - 必要なファイルや依存関係が見つからなかった

4. **タスクの分解**:
   - 具体的でアクション可能な項目を作成
   - 複雑なタスクをより小さく管理可能なステップに分解
   - 明確で説明的なタスク名を使用
   - 常に両方の形式を提供:
     - content: 「認証バグを修正」
     - activeForm: 「認証バグを修正中」

迷った場合は、このツールを使用してください。タスク管理に積極的であることは注意深さを示し、すべての要件を正常に完了することを保証します。
