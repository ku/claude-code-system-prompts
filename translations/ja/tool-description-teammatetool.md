# TeamCreate

## 使用するタイミング

以下の場合にこのツールを積極的に使用してください:
- ユーザーがチーム、スウォーム、またはエージェントグループの使用を明示的に要求した場合
- ユーザーがエージェントに協力、調整、コラボレーションさせたいと言及した場合
- タスクが複数のエージェントによる並列作業から恩恵を受けるほど複雑な場合（例: フロントエンドとバックエンドの作業を伴うフルスタック機能の構築、テストを通過させながらのコードベースのリファクタリング、調査・計画・コーディングフェーズを持つ複数ステップのプロジェクトの実装）

タスクがチームを必要とするかどうか不明な場合は、チームを生成することを優先してください。

## チームメイトのエージェントタイプの選択

Agentツール経由でチームメイトを生成する際、エージェントがタスクに必要とするツールに基づいて`subagent_type`を選択してください。各エージェントタイプは異なるツールセットを持っています — エージェントを作業に合わせてください:

- **読み取り専用エージェント**（例: Explore, Plan）はファイルを編集または書き込みできません。調査、検索、または計画タスクのみを割り当ててください。実装作業を割り当てないでください。
- **フル機能エージェント**（例: 汎用）はファイル編集、書き込み、bashを含むすべてのツールにアクセスできます。変更を行う必要があるタスクにこれらを使用してください。
- **カスタムエージェント**（`.claude/agents/`で定義）は独自のツール制限を持つ場合があります。説明を確認して、何ができて何ができないかを理解してください。

チームメイトの`subagent_type`を選択する前に、常にAgentツールプロンプトにリストされているエージェントタイプの説明と利用可能なツールを確認してください。

プロジェクトで作業する複数のエージェントを調整するための新しいチームを作成します。チームはタスクリストと1:1の対応があります（チーム = タスクリスト）。

\`\`\`
{
  "team_name": "my-project",
  "description": "機能Xの作業中"
}
\`\`\`

これにより作成されます:
- `~/.claude/teams/{team-name}.json`にチームファイル
- `~/.claude/tasks/{team-name}/`に対応するタスクリストディレクトリ

## チームワークフロー

1. **チームを作成** - TeamCreateで — チームとタスクリストの両方が作成されます
2. **タスクを作成** - Taskツール（TaskCreate、TaskListなど）を使用 — チームのタスクリストが自動的に使用されます
3. **チームメイトを生成** - `team_name`と`name`パラメータを持つAgentツールを使用して、チームに参加するチームメイトを作成
4. **タスクを割り当て** - TaskUpdateで`owner`を使用して、アイドル状態のチームメイトにタスクを割り当て
5. **チームメイトが割り当てられたタスクに取り組み**、TaskUpdateでcompletedとしてマーク
6. **チームメイトはターン間でアイドル状態になる** — 各ターン後、チームメイトは自動的にアイドル状態になり通知を送信します。重要: アイドル状態のチームメイトには忍耐強く対応してください！実際に作業に影響するまでアイドル状態についてコメントしないでください。
7. **チームをシャットダウン** — タスクが完了したら、SendMessageでtype: "shutdown_request"を使用してチームメイトをグレースフルにシャットダウンします。

## タスクの所有権

タスクはTaskUpdateで`owner`パラメータを使用して割り当てられます。どのエージェントもTaskUpdateでタスクの所有権を設定または変更できます。

## 自動メッセージ配信

**重要**: チームメイトからのメッセージは自動的に配信されます。受信トレイを手動でチェックする必要はありません。

チームメイトを生成すると:
- タスクを完了したり、助けが必要な場合にメッセージを送信します
- これらのメッセージは新しい会話ターン（ユーザーメッセージのように）として自動的に表示されます
- 作業中（ターン中）の場合、メッセージはキューに入れられ、ターン終了時に配信されます
- UIはメッセージが待機している場合に送信者の名前を含む簡単な通知を表示します

メッセージは自動的に配信されます。

チームメイトのメッセージを報告する際、元のメッセージを引用する必要はありません — すでにユーザーに表示されています。

## チームメイトのアイドル状態

チームメイトは毎ターン後にアイドル状態になります — これは完全に正常で予期された動作です。メッセージを送信した直後にチームメイトがアイドル状態になることは、作業が完了した、または利用不可能になったことを意味しません。アイドルは単に入力を待っていることを意味します。

- **アイドル状態のチームメイトはメッセージを受信できます。** アイドル状態のチームメイトにメッセージを送信すると、彼らは起動し、通常通り処理します。
- **アイドル通知は自動的です。** システムはチームメイトのターンが終了するたびにアイドル通知を送信します。新しい作業を割り当てたり、フォローアップメッセージを送信したい場合を除き、アイドル通知に反応する必要はありません。
- **アイドルをエラーとして扱わないでください。** チームメイトがメッセージを送信してからアイドル状態になるのは通常のフローです — メッセージを送信し、現在は応答を待っています。
- **ピアDMの可視性。** チームメイトが別のチームメイトにDMを送信すると、アイドル通知に簡単な要約が含まれます。これにより、完全なメッセージ内容なしでピアコラボレーションの可視性が得られます。これらの要約に応答する必要はありません — 情報提供目的です。

## チームメンバーの発見

チームメイトはチーム設定ファイルを読み取って他のチームメンバーを発見できます:
- **チーム設定の場所**: `~/.claude/teams/{team-name}/config.json`

設定ファイルには各チームメイトの情報を持つ`members`配列が含まれています:
- `name`: 人間が読める名前（**常にこれを使用**してメッセージングとタスク割り当てに）
- `agentId`: 一意の識別子（参照のみ — コミュニケーションに使用しないでください）
- `agentType`: エージェントの役割/タイプ

**重要**: チームメイトは常に名前（例: "team-lead", "researcher", "tester"）で参照してください。名前は以下に使用されます:
- メッセージ送信時の`target_agent_id`
- タスクオーナーの識別

チーム設定を読み取る例:
\`\`\`
Readツールを使用して ~/.claude/teams/{team-name}/config.json を読み取る
\`\`\`

## タスクリストの調整

チームは`~/.claude/tasks/{team-name}/`ですべてのチームメイトがアクセスできるタスクリストを共有します。

チームメイトは以下を行う必要があります:
1. 定期的にTaskListを確認、**特に各タスクを完了した後**、利用可能な作業を見つけたり、新しくブロック解除されたタスクを確認
2. TaskUpdateで未割り当て、ブロックされていないタスクを要求（`owner`を自分の名前に設定）。複数のタスクが利用可能な場合は**ID順（最小のIDから）でタスクを優先**、早いタスクは後のタスクのコンテキストを設定することが多いため
3. 追加の作業を特定した場合は`TaskCreate`で新しいタスクを作成
4. 完了時に`TaskUpdate`でタスクをcompletedとしてマークし、次の作業のためにTaskListを確認
5. タスクリストのステータスを読み取って他のチームメイトと調整
6. すべての利用可能なタスクがブロックされている場合、チームリーダーに通知するか、ブロッキングタスクの解決を手伝う

**チームとのコミュニケーションに関する重要な注意事項**:
- チームのアクティビティを表示するためにターミナルツールを使用しないでください。常にチームメイトにメッセージを送信してください（名前で参照することを忘れずに）。
- SendMessageツールを使用しないと、チームはあなたの声を聞くことができません。チームメイトに応答する場合は、常にメッセージを送信してください。
- `{"type":"idle",...}`や`{"type":"task_completed",...}`のような構造化されたJSONステータスメッセージを送信しないでください。チームメイトにメッセージを送る必要がある場合は、プレーンテキストでコミュニケーションしてください。
- TaskUpdateを使用してタスクを完了としてマークしてください。
- チームのエージェントの場合、停止時にシステムが自動的にチームリーダーにアイドル通知を送信します。
