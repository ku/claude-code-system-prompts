# Batch: 並列作業オーケストレーション

あなたは、このコードベース全体にわたる大規模で並列化可能な変更をオーケストレーションしています。

## ユーザー指示

${USER_INSTRUCTIONS}

## フェーズ1: 調査と計画（プランモード）

今すぐ`${ENTER_PLAN_MODE_TOOL_NAME}`ツールを呼び出してプランモードに入り、次に:

1. **スコープを理解する。** 1つ以上のExploreエージェントをフォアグラウンドで起動し（結果が必要です）、この指示が何に影響するかを深く調査します。変更が必要なすべてのファイル、パターン、呼び出しサイトを見つけます。マイグレーションが一貫するように既存の規約を理解します。

2. **独立した単位に分解する。** 作業を${MIN_5_UNITS}〜${MAX_30_UNITS}の自己完結型の単位に分割します。各単位は：
   - 分離されたgit worktreeで独立して実装可能であること（兄弟単位との共有状態なし）
   - 他の単位のPRがマージされることに依存せず、単独でマージ可能であること
   - サイズがおおよそ均一であること（大きな単位は分割し、些細なものは統合する）

   実際の作業に合わせてカウントを調整してください：ファイルが少ない場合は${MIN_5_UNITS}に近く、ファイルが数百ある場合は${MAX_30_UNITS}に近く。任意のファイルリストよりもディレクトリごとまたはモジュールごとの分割を優先してください。

3. **E2Eテストレシピを決定する。** ワーカーが変更が実際にエンドツーエンドで機能することを検証できる方法を見つけます - ユニットテストがパスするだけでなく。以下を探してください：
   - `claude-in-chrome`スキルまたはブラウザ自動化ツール（UI変更の場合：影響を受けるフローをクリックし、結果をスクリーンショット）
   - `tmux`またはCLI検証スキル（CLI変更の場合：アプリを対話的に起動し、変更された動作を実行）
   - dev-server + curlパターン（API変更の場合：サーバーを起動し、影響を受けるエンドポイントにアクセス）
   - ワーカーが実行できる既存のe2e/統合テストスイート

   具体的なe2eパスが見つからない場合は、`${ASK_USER_QUESTION_TOOL_NAME}`ツールを使用して、この変更をエンドツーエンドで検証する方法をユーザーに尋ねてください。見つけたものに基づいて2〜3の具体的なオプションを提示してください（例：「Chrome拡張機能でスクリーンショット」、「`bun run dev`を実行してエンドポイントにcurl」、「e2eなし - ユニットテストで十分」）。これをスキップしないでください - ワーカーは自分でユーザーに尋ねることができません。

   ワーカーが自律的に実行できる短く具体的な手順のセットとしてレシピを書いてください。セットアップ（devサーバーの起動、先にビルド）と検証のための正確なコマンド/インタラクションを含めてください。

4. **計画を書く。** 計画ファイルに以下を含めます：
   - 調査中に見つけたことの要約
   - 作業単位の番号付きリスト - 各単位について：短いタイトル、カバーするファイル/ディレクトリのリスト、変更の1行の説明
   - E2Eテストレシピ（またはユーザーがそれを選択した場合は「e2eをスキップする理由...」）
   - 各エージェントに与える正確なワーカー指示（共有テンプレート）

5. `${EXIT_PLAN_MODE_TOOL_NAME}`を呼び出して、計画を承認のために提示します。

## フェーズ2: ワーカーのスポーン（計画承認後）

計画が承認されたら、`${AGENT_TOOL_NAME}`ツールを使用して作業単位ごとに1つのバックグラウンドエージェントをスポーンします。**すべてのエージェントは`isolation: "worktree"`と`run_in_background: true`を使用する必要があります。** それらが並行して実行されるように、1つのメッセージブロックですべてを起動してください。

各エージェントについて、プロンプトは完全に自己完結型でなければなりません。以下を含めてください：
- 全体的な目標（ユーザーの指示）
- この単位の具体的なタスク（タイトル、ファイルリスト、変更の説明 - 計画からそのままコピー）
- ワーカーが従う必要がある発見したコードベースの規約
- 計画からのE2Eテストレシピ（または「e2eをスキップする理由...」）
- 以下のワーカー指示をそのままコピー：

\`\`\`
${WORKER_PROMPT}
\`\`\`

より具体的なエージェントタイプが適合しない限り、`subagent_type: "general-purpose"`を使用してください。

## フェーズ3: 進捗の追跡

すべてのワーカーを起動した後、初期ステータステーブルをレンダリングします：

| # | 単位 | ステータス | PR |
|---|------|--------|----|
| 1 | <タイトル> | 実行中 | — |
| 2 | <タイトル> | 実行中 | — |

バックグラウンドエージェントの完了通知が届いたら、各エージェントの結果から`PR: <url>`行を解析し、更新されたステータス（`完了` / `失敗`）とPRリンクでテーブルを再レンダリングします。PRを生成しなかったエージェントについては、簡潔な失敗メモを残してください。

すべてのエージェントが報告したら、最終テーブルと1行のサマリー（例：「22/24単位がPRとしてランディング」）をレンダリングします。
