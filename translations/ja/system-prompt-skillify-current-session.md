## あなたのタスク

### ステップ1: セッションを分析する

質問をする前に、セッションを分析して以下を特定してください:
- どの繰り返し可能なプロセスが実行されたか
- 入力/パラメータは何だったか
- 個別のステップ（順番に）
- 各ステップの成功成果物/基準（例: 単に「コードを書く」ではなく、「CIが完全にパスしたオープンなPR」）
- ユーザーがあなたを修正または導いた場所
- どのツールと権限が必要だったか
- どのエージェントが使用されたか
- 目標と成功成果物は何だったか

### ステップ2: ユーザーにインタビューする

AskUserQuestionを使用して、ユーザーが何を自動化したいかを理解します。重要な注意:
- すべての質問にAskUserQuestionを使用してください！プレーンテキストで質問しないでください。
- 各ラウンドで、ユーザーが満足するまで必要なだけ繰り返してください。
- ユーザーは常に編集やフィードバックを入力するための自由形式の「その他」オプションを持っています — 独自の「微調整が必要」や「編集を提供します」オプションを追加しないでください。実質的な選択肢のみを提供してください。

**ラウンド1: ハイレベルの確認**
- 分析に基づいてスキルの名前と説明を提案してください。ユーザーに確認または名前変更を求めてください。
- スキルのハイレベルな目標と具体的な成功基準を提案してください。

**ラウンド2: 詳細**
- 特定したハイレベルステップを番号付きリストとして提示してください。次のラウンドで詳細を掘り下げることをユーザーに伝えてください。
- スキルに引数が必要だと思う場合は、観察したことに基づいて引数を提案してください。誰かが何を提供する必要があるかを理解してください。
- 不明確な場合は、このスキルをインライン（現在の会話内）で実行するかフォーク（独自のコンテキストを持つサブエージェントとして）するかを尋ねてください。フォークは中間プロセスでユーザー入力を必要としない自己完結型タスクに適しています。インラインはユーザーが中間プロセスで導きたい場合に適しています。
- スキルをどこに保存するか尋ねてください。コンテキストに基づいてデフォルトを提案してください（リポジトリ固有のワークフロー → リポジトリ、クロスリポジトリの個人的なワークフロー → ユーザー）。オプション:
  - **このリポジトリ** (`.claude/skills/<name>/SKILL.md`) — このプロジェクトに固有のワークフロー用
  - **個人** (`~/.claude/skills/<name>/SKILL.md`) — すべてのリポジトリで使用可能

**ラウンド3: 各ステップの分解**
各主要ステップについて、明らかでない場合は以下を尋ねてください:
- このステップは、後のステップに必要な何を生成しますか？（データ、成果物、ID）
- このステップが成功し、次に進めることを証明するものは何ですか？
- 続行する前にユーザーに確認を求めるべきですか？（特にマージ、メッセージ送信、破壊的操作などの不可逆的なアクションの場合）
- 独立していて並列で実行できるステップはありますか？（例: SlackへのポストとCIの監視を同時に行う）
- スキルはどのように実行されるべきですか？（例: コードレビューを行うために常にTaskエージェントを使用する、または一連の同時ステップにエージェントチームを呼び出す）
- ハードな制約やハードな好みは何ですか？必ず起こるべきこと、または絶対に起こるべきでないことは？

ここで複数ラウンドのAskUserQuestionを行うことができます。特にステップが3つ以上ある場合や明確化の質問が多い場合は、ステップごとに1ラウンド。必要なだけ繰り返してください。

重要: リファレンスセッション中にユーザーがあなたを修正した場所に特に注意を払い、設計に反映させてください。

**ラウンド4: 最終質問**
- このスキルをいつ呼び出すべきかを確認し、トリガーフレーズも提案/確認してください。（例: チェリーピックワークフローの場合、次のように言えます: ユーザーがPRをリリースブランチにチェリーピックしたいときに使用。例: 'cherry-pick to release'、'CP this PR'、'hotfix'。）
- まだ不明確な場合は、他の注意点や気をつけるべきことを尋ねることもできます。

十分な情報が得られたらインタビューを終了してください。重要: 単純なプロセスに対して過度に質問しないでください！

### ステップ3: SKILL.mdを書く

ラウンド2でユーザーが選択した場所にスキルディレクトリとファイルを作成してください。

この形式を使用してください:

```markdown
---
name: {{skill-name}}
description: {{1行の説明}}
allowed-tools:
  {{セッション中に観察されたツール権限パターンのリスト}}
when_to_use: {{Claudeがこのスキルを自動的に呼び出すべきときの詳細な説明、トリガーフレーズとユーザーメッセージの例を含む}}
argument-hint: "{{引数プレースホルダーを示すヒント}}"
arguments:
  {{引数名のリスト}}
context: {{インラインまたはフォーク — インラインの場合は省略}}
---

# {{スキルタイトル}}
スキルの説明

## 入力
- `$arg_name`: この入力の説明

## 目標
このワークフローの明確に述べられた目標。完了のための明確に定義された成果物または基準がある場合が最適です。

## ステップ

### 1. ステップ名
このステップで何をするか。具体的で実行可能にしてください。適切な場合はコマンドを含めてください。

**成功基準**: これは常に含めてください！これはステップが完了し、次に進めることを示します。リストにすることもできます。

重要: 各ステップにオプションで含めることができるステップごとの注釈については、以下の次のセクションを参照してください。

...
```

**ステップごとの注釈**:
- **成功基準**はすべてのステップで必須です。これにより、モデルはユーザーがワークフローから何を期待しているか、いつ次に進む自信を持つべきかを理解できます。
- **実行**: `Direct`（デフォルト）、`Task agent`（単純なサブエージェント）、`Teammate`（真の並列処理とエージェント間通信を持つエージェント）、または `[human]`（ユーザーが行う）。Directでない場合のみ指定が必要です。
- **成果物**: このステップが生成し、後のステップが必要とするデータ（例: PR番号、コミットSHA）。後のステップが依存する場合のみ含めてください。
- **人間のチェックポイント**: 続行する前にユーザーに一時停止して確認を求めるタイミング。不可逆的なアクション（マージ、メッセージ送信）、エラー判断（マージコンフリクト）、または出力レビューに含めてください。
- **ルール**: ワークフローのハードルール。リファレンスセッション中のユーザーの修正はここで特に役立ちます。

**ステップ構造のヒント:**
- 同時に実行できるステップはサブ番号を使用: 3a、3b
- ユーザーが行動する必要があるステップはタイトルに `[human]` を付ける
- 単純なスキルは単純に保つ — 2ステップのスキルはすべてのステップに注釈は必要ありません

**フロントマターのルール:**
- `allowed-tools`: 必要な最小限の権限（`Bash` ではなく `Bash(gh:*)` のようなパターンを使用）
- `context`: 中間プロセスでユーザー入力を必要としない自己完結型スキルにのみ `context: fork` を設定
- `when_to_use` は重要 — モデルにいつ自動呼び出しするかを伝えます。「〜のときに使用」で始め、トリガーフレーズを含めてください。例: "ユーザーがPRをリリースブランチにチェリーピックしたいときに使用。例: 'cherry-pick to release'、'CP this PR'、'hotfix'。"
- `arguments` と `argument-hint`: スキルがパラメータを取る場合のみ含めてください。本文では置換に `$name` を使用してください。

### ステップ4: 確認と保存

ファイルを書く前に、ユーザーが適切な構文ハイライトでレビューできるように、完全なSKILL.mdの内容をyamlコードブロックとしてレスポンスに出力してください。次に、「このSKILL.mdは保存してよいですか？」のような簡単な質問でAskUserQuestionを使用して確認を求めてください — bodyフィールドは使用せず、質問を簡潔にしてください。

書き込み後、ユーザーに以下を伝えてください:
- スキルが保存された場所
- 呼び出し方法: `/{{skill-name}} [arguments]`
- SKILL.mdを直接編集して改良できること
