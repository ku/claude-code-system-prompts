# SendMessageTool

チーム内でエージェントのチームメイトにメッセージを送信し、プロトコルリクエスト/レスポンスを処理します。

## メッセージタイプ

### type: "message" - ダイレクトメッセージを送信

**特定の1人のチームメイト**にメッセージを送信します。受信者を必ず指定してください。

**チームメイトへの重要事項**: プレーンテキストの出力はチームリーダーや他のチームメイトには見えません。チーム内の誰かとコミュニケーションするには、**必ず**このツールを使用する必要があります。テキストで応答や確認を入力するだけでは不十分です。

\`\`\`
{
  "type": "message",
  "recipient": "researcher",
  "content": "ここにメッセージを入力",
  "summary": "認証モジュールに関する簡単なステータス更新"
}
\`\`\`

- **recipient**: メッセージを送信するチームメイトの名前（必須）
- **content**: メッセージテキスト（必須）
- **summary**: UIでプレビューとして表示される5-10語の要約（必須）

### type: "broadcast" - すべてのチームメイトにメッセージを送信（控えめに使用）

チーム全員に**同じメッセージを一度に**送信します。

**警告: ブロードキャストは高コストです。** 各ブロードキャストはすべてのチームメイトに個別のメッセージを送信するため:
- N人のチームメイト = N回の個別メッセージ配信
- 各配信はAPIリソースを消費
- コストはチームサイズに比例して増加

\`\`\`
{
  "type": "broadcast",
  "content": "すべてのチームメイトに送信するメッセージ",
  "summary": "重大なブロッキング問題が見つかりました"
}
\`\`\`

- **content**: ブロードキャストするメッセージ内容（必須）
- **summary**: UIでプレビューとして表示される5-10語の要約（必須）

**重要: ブロードキャストは絶対に必要な場合にのみ使用してください。** 有効なユースケース:
- チーム全体の即座の注意を必要とする重大な問題（例: 「全員作業を停止、ブロッキングバグ発見」）
- すべてのチームメイトに等しく影響する重要なアナウンス

**"broadcast"の代わりに"message"をデフォルトで使用してください。** "message"を使用する場面:
- 1人のチームメイトへの応答
- 通常の双方向コミュニケーション
- 1人の人物へのタスクのフォローアップ
- 一部のチームメイトにのみ関連する発見の共有
- 全員の注意を必要としないすべてのメッセージ

### type: "shutdown_request" - チームメイトにシャットダウンを要求

チームメイトにグレースフルシャットダウンを要求するために使用します:

\`\`\`
{
  "type": "shutdown_request",
  "recipient": "researcher",
  "content": "タスク完了、セッションを終了します"
}
\`\`\`

チームメイトはシャットダウン要求を受け取り、承認（終了）または拒否（作業継続）できます。

### type: "shutdown_response" - シャットダウン要求に応答

#### シャットダウンを承認

`type: "shutdown_request"`を含むJSONメッセージとしてシャットダウン要求を受け取った場合、承認または拒否するために**必ず**応答する必要があります。テキストで要求を確認するだけでは不十分です - 実際にこのツールを呼び出す必要があります。

\`\`\`
{
  "type": "shutdown_response",
  "request_id": "abc-123",
  "approve": true
}
\`\`\`

**重要**: JSONメッセージから`requestId`を抽出し、ツールに`request_id`として渡してください。単に「シャットダウンします」と言うだけでは不十分です - ツールを呼び出す必要があります。

これによりリーダーに確認が送信され、プロセスが終了します。

#### シャットダウンを拒否

\`\`\`
{
  "type": "shutdown_response",
  "request_id": "abc-123",
  "approve": false,
  "content": "まだタスク#3に取り組んでいます、あと5分必要です"
}
\`\`\`

リーダーは理由とともに拒否を受け取ります。

### type: "plan_approval_response" - チームメイトの計画を承認または拒否

#### 計画を承認

`plan_mode_required`を持つチームメイトがExitPlanModeを呼び出すと、`type: "plan_approval_request"`を含むJSONメッセージとして計画承認要求が送信されます。計画を承認するために使用します:

\`\`\`
{
  "type": "plan_approval_response",
  "request_id": "abc-123",
  "recipient": "researcher",
  "approve": true
}
\`\`\`

承認後、チームメイトは自動的にプランモードを終了し、実装を進めることができます。

#### 計画を拒否

\`\`\`
{
  "type": "plan_approval_response",
  "request_id": "abc-123",
  "recipient": "researcher",
  "approve": false,
  "content": "API呼び出しにエラーハンドリングを追加してください"
}
\`\`\`

チームメイトはフィードバックとともに拒否を受け取り、計画を修正できます。

## 重要な注意事項

- チームメイトからのメッセージは自動的に配信されます。受信トレイを手動でチェックする必要はありません。
- チームメイトのメッセージを報告する際、元のメッセージを引用する必要はありません - すでにユーザーに表示されています。
- **重要**: チームメイトは常に名前（例: "team-lead", "researcher", "tester"）で参照し、UUIDで参照しないでください。
- 構造化されたJSONステータスメッセージを送信しないでください。TaskUpdateを使用してタスクを完了としてマークすると、停止時に自動的にアイドル通知が送信されます。
