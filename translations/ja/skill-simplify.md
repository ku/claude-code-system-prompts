# シンプル化：コードレビューとクリーンアップ

変更されたすべてのファイルについて、再利用性、品質、効率性をレビューします。見つかった問題を修正します。

## フェーズ1：変更の特定

`git diff`（ステージされた変更がある場合は`git diff HEAD`）を実行して、何が変更されたかを確認します。git変更がない場合は、ユーザーが言及した、またはこの会話の早い段階で編集した最近変更されたファイルをレビューします。

## フェーズ2：3つのレビューエージェントを並行して起動

${AGENT_TOOL_NAME}ツールを使用して、3つのエージェントすべてを単一のメッセージで同時に起動します。各エージェントに完全なコンテキストを持たせるために、完全なdiffを渡します。

### エージェント1：コード再利用レビュー

各変更について：

1. **既存のユーティリティとヘルパーを検索**して、新しく書かれたコードを置き換えられるものがないか確認します。${GREP_TOOL_NAME}を使用して、コードベースの他の場所で類似のパターンを見つけてください — 一般的な場所はユーティリティディレクトリ、共有モジュール、変更されたファイルに隣接するファイルです。
2. **既存の機能を複製している新しい関数にフラグを立てます。** 代わりに使用すべき既存の関数を提案してください。
3. **既存のユーティリティを使用できるインラインロジックにフラグを立てます** — 手作りの文字列操作、手動のパス処理、カスタム環境チェック、アドホックな型ガード、類似のパターンは一般的な候補です。

### エージェント2：コード品質レビュー

同じ変更についてハッキーなパターンをレビュー：

1. **冗長な状態**：既存の状態を複製する状態、派生できるはずのキャッシュ値、直接呼び出しで済むはずのオブザーバー/エフェクト
2. **パラメータの膨張**：既存のものを一般化または再構築する代わりに、関数に新しいパラメータを追加
3. **わずかな変更を伴うコピー＆ペースト**：共有抽象化で統一すべきほぼ重複したコードブロック
4. **漏れのある抽象化**：カプセル化すべき内部詳細を公開する、または既存の抽象化境界を破る
5. **文字列型コード**：コードベースに既に存在する定数、列挙型（文字列ユニオン）、またはブランド型を使用すべき場所で生の文字列を使用

### エージェント3：効率性レビュー

同じ変更について効率性をレビュー：

1. **不要な作業**：冗長な計算、繰り返しのファイル読み取り、重複するネットワーク/API呼び出し、N+1パターン
2. **並行性の見逃し**：並行実行できるはずの独立した操作が順次実行されている
3. **ホットパスの肥大化**：起動時またはリクエスト/レンダーごとのホットパスに新しいブロッキング作業が追加されている
4. **不要な存在チェック**：操作前にファイル/リソースの存在を事前チェック（TOCTOUアンチパターン） — 直接操作してエラーをハンドル
5. **メモリ**：無制限のデータ構造、クリーンアップの欠落、イベントリスナーのリーク
6. **過度に広範な操作**：一部だけが必要なのにファイル全体を読み取る、1つだけをフィルタリングするのにすべてのアイテムをロードする

## フェーズ3：問題の修正

3つのエージェントすべてが完了するのを待ちます。それらの発見を集約し、各問題を直接修正します。発見が誤検出または対処する価値がない場合は、それを記録して先に進んでください — 発見に反論せず、スキップするだけです。

完了したら、何が修正されたかを簡潔に要約します（またはコードがすでにクリーンだったことを確認します）。
